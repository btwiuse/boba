# Boba v0.3: Three-Layer Middleware Architecture

## Goal

Add composable middleware to the `serve` package across three distinct layers — handshake, session I/O, and handler — so that the `net/http` ecosystem (auth, rate limiting, observability) is reusable at the boundary, terminal-protocol-aware concerns (clipboard gating, paste limits, idle timeouts) become first-class, and BubbleTea handlers can be wrapped Wish-style. Also add three always-on resource-protection config knobs that the v0.4 built-in middleware will rely on.

This spec implements the architecture proposed in `docs/DESIGN_MIDDLEWARE.md`, with all open questions resolved.

## Design Principles

- **Three layers, not one.** HTTP, WebTransport, Sip, and VT each have a different natural object to wrap. A single middleware type would either exclude legitimate concerns or mislead users about what `net/http` can do for them.
- **Single chain of execution per layer.** Built-in behaviors (auth, connection limits) become middleware so the chain is the source of truth — no special-cased branches in the WS/WT handlers.
- **Functional options for behavior, `Config` for data.** Anything someone might marshal from JSON/env stays on `Config`; anything that's a function value uses `serve.With*(...)` options on `NewServer`.
- **Outermost-first composition.** The first middleware listed sees calls first on the way in (and last on the way out). Matches `net/http` and Wish.
- **YAGNI over completeness.** No `ProgramMiddleware`, no parallel APIs, no helper base structs that the embed-an-interface idiom already covers. We're pre-1.0 with no backwards-compatibility concerns; redundant API surface gets cut.

## Public API Surface

### Middleware types

```go
// Layer 1 — handshake. Runs on both WS upgrade and WT CONNECT, on the raw
// *http.Request before the upgrade is performed. Return non-nil to reject;
// return *ConnectError for control over the rejection response. Middleware
// that wants to add request-scoped values (e.g., identity) does so via
// r = r.WithContext(ctx) before calling next.
type ConnectHandler    func(r *http.Request) error
type ConnectMiddleware func(next ConnectHandler) ConnectHandler

// Layer 2 — session I/O. Wraps the Session returned by SessionFactory.
// Composes via Go's interface-embedding idiom.
type SessionMiddleware func(Session) Session

// Layer 3 — handler. Matches Wish's bubbletea.Middleware shape.
type Middleware func(next Handler) Handler
```

`Handler` retains the v0.2 shape: `func(Session) (tea.Model, []tea.ProgramOption)`.

### ConnectError

```go
type ConnectError struct {
    Status  int          // HTTP status; mapped to QUIC error code on WT
    Headers http.Header  // optional response headers (e.g. WWW-Authenticate)
    Body    string       // optional response body (WS path only)
    WTCode  uint32       // optional override of the default WT mapping
    Cause   error        // optional underlying error for logging
}
func (e *ConnectError) Error() string
func (e *ConnectError) Unwrap() error
```

- A plain `error` returned from `ConnectHandler` is treated as `&ConnectError{Status: 500}` (i.e., 500 on WS, default mapping on WT).
- Default WS→WT mapping when `WTCode == 0`: 4xx → `0x01`, 5xx → `0x02`, otherwise `0x00`.
- A package-private sentinel `errResponseWritten` signals "the response was already written by lifted HTTP middleware; framework, do not write again." Returned by `LiftHTTPMiddleware` when the wrapped middleware writes to `ResponseWriter` instead of calling `next`.

### LiftHTTPMiddleware

```go
func LiftHTTPMiddleware(mw func(http.Handler) http.Handler) ConnectMiddleware
```

Adapts standard `net/http` middleware (chi, gorilla, otelhttp, prometheus-handler, tollbooth, etc.) to `ConnectMiddleware`. The lifted middleware sees a real `http.ResponseWriter` and `*http.Request`. Two outcomes:

1. **Lifted middleware calls `next(w, r)` and lets it through:** the adapter invokes the boba `ConnectHandler` chain, returns its result.
2. **Lifted middleware writes a response and does not call `next`:** the adapter returns `errResponseWritten`. The framework treats this as "rejected, response already sent" and stops.

The lifted middleware MUST NOT assume it can inspect or rewrite the response after `next` returns — there is no response after upgrade.

### Installation API (functional options on NewServer)

```go
type Option func(*Server)

func WithConnectMiddleware(mw ...ConnectMiddleware) Option
func WithSessionMiddleware(mw ...SessionMiddleware) Option
func WithMiddleware(mw ...Middleware) Option
func WithSessionFactory(f SessionFactory) Option

func NewServer(cfg Config, opts ...Option) *Server
```

Semantics:
- `With*Middleware` are variadic-additive: multiple calls append, single call with multiple values appends in order.
- `WithSessionFactory` is last-wins; default is `defaultSessionFactory`.
- `Server` is immutable post-construction; the only post-construction state mutation is `Serve(ctx, handler)`.

### Identity propagation

```go
type Identity interface {
    String() string
}

func WithIdentity(ctx context.Context, id Identity) context.Context
func IdentityFromContext(ctx context.Context) (Identity, bool)
```

Lives in the `serve` package. Layer-1 middleware writes identity into the request context; the framework propagates that context onto the Session (`Session.Context()` returns it); layer 2/3 read via `IdentityFromContext`. The interface stays minimal so v0.4+ extensions (claims, roles) are additive.

### Config knobs

```go
type Config struct {
    // ... existing fields ...

    // MaxPasteBytes caps the size of Sip bracketed-paste payloads.
    // Zero means default (1 MiB).
    MaxPasteBytes int

    // ResizeThrottle coalesces rapid inbound resize messages.
    // Zero means default (16ms).
    ResizeThrottle time.Duration

    // MaxWindowDims rejects resize messages exceeding these dimensions.
    // Zero value (Width == 0 || Height == 0) means default (4096x4096).
    MaxWindowDims WindowSize
}

func ConfigFromContext(ctx context.Context) Config
```

`ConfigFromContext` returns `Config` by value (an immutable read). Available at every layer: layer-1 middleware sees it via the request context (the framework decorates the context before invoking the chain); layer-2 and layer-3 see it via `Session.Context()`.

### Deletions

The following are removed in this work (no backwards-compatibility concerns pre-1.0):

- `func (s *Server) SetSessionFactory(factory SessionFactory)` — replaced by `WithSessionFactory`.
- `func (s *Server) ServeWithProgram(ctx context.Context, handler ProgramHandler) error`.
- `type ProgramHandler func(sess Session) *tea.Program`.
- The `progHandler` field on `Server`.
- `runBubbleTeaProgram`.
- The `progHandler` branch in `Server.runSession`.

Single execution path is `Server.Serve(ctx, Handler)` → `runBubbleTea`.

## Architecture

### Composition order

All three middleware types compose **outermost-first**: `mw[0]` sees the call first on the way in and last on the way out. Concretely:

```
chain = mw[0](mw[1](mw[2](base)))
```

For layer 1 (callable chain): `chain(ctx, r)` invokes `mw[0]` first; each middleware decides whether to call `next` and whether to return early.

For layers 2 and 3 (wrap-and-delegate): the framework constructs the wrapped value once at install time. `mw[0]` is the outermost wrapper, sees calls first on the way in to the underlying object, and sees the returned value last on the way out.

### Layer 1 — connect chain

The Server holds an ordered slice of `ConnectMiddleware`. At install time:

1. The user's `WithConnectMiddleware(...)` calls populate the slice in install order.
2. The Server **appends** built-in connect middleware so they run innermost (last in the chain):
   - **Basic auth** (only installed if `cfg.BasicUsername != ""`): rewritten from today's `checkAuth` to return `&ConnectError{Status: 401, Headers: http.Header{"WWW-Authenticate": []string{"Basic realm=\"boba\""}}, Body: "Unauthorized"}` on failure.
   - **Connection limit** (always installed): rewritten from today's `tryAcquireConnection` to return `&ConnectError{Status: 503, Body: "max connections reached"}` when the limit is hit. Acquisition happens on success; release happens via a deferred handler in the WS/WT path.

The terminal `ConnectHandler` is a no-op `func(r *http.Request) error { return nil }`. Middleware that wants to reject does so by returning before calling `next`.

The framework decorates the request context with the `Config` value (via `ConfigFromContext`-readable key) before invoking the chain.

### Layer 2 — session middleware chain

After `createSession` returns a bare `Session`, the framework wraps it:

```go
sess := factory(ctx, size)
for i := len(s.sessionMW) - 1; i >= 0; i-- {
    sess = s.sessionMW[i](sess)
}
```

Iteration is reverse so that `mw[0]` ends up outermost. Middleware authors use the embed-an-interface idiom:

```go
type osc52gate struct {
    serve.Session
    mode string
}

func (g *osc52gate) OutputReader() io.Reader {
    return scanReader(g.Session.OutputReader(), g.mode)
}
```

Lifecycle middleware (activeterm, idletimeout) spawn a goroutine in their constructor that watches `sess.Done()` / `sess.Context()` and calls `sess.Close()` when their condition fires. `Session.Close()` is documented as **idempotent**: subsequent calls return nil after the first. Middleware that holds resources overrides `Close()` to release them, then delegates to `g.Session.Close()`.

### Layer 3 — handler middleware chain

Applied once at install time:

```go
h := userHandler
for i := len(s.middleware) - 1; i >= 0; i-- {
    h = s.middleware[i](h)
}
s.handler = h
```

`runBubbleTea` invokes `s.handler(sess)` per session.

### Handler refactor

`handleWS` and `handleWT` collapse to roughly the same shape:

```
1. Run the connect chain on r (request context decorated with Config).
2. If chain returned non-nil error:
     - WS path: write *ConnectError fields (Status, Headers, Body) to ResponseWriter, return.
     - WT path: close with WTCode (or default mapping from Status), return.
     - errResponseWritten: return without writing.
3. Upgrade (websocket.Accept / wtServer.Upgrade).
4. Read initial Resize. Validate Cols/Rows > 0 and against Config.MaxWindowDims.
5. createSession via factory.
6. Wrap Session with session-middleware chain.
7. Start runSession in a goroutine (calls layer-3-wrapped handler).
8. Enter transport message loop with paste-cap and resize-throttle enforced.
```

Both handlers still differ in transport mechanics but share steps 1, 2, 4, 5, 6 — opportunity for a small shared helper, judgment call at implementation time. Step 8's enforcement of `MaxPasteBytes` and `ResizeThrottle` lives in `handleWebSocket` / `handleWebTransport` (the existing message-loop functions).

### Identity propagation flow

```
Layer-1 middleware:
   r = r.WithContext(serve.WithIdentity(r.Context(), id))
   return next(r)

Framework (after layer-1 chain succeeds):
   sessCtx is derived from r.Context() — identity inherited.

Layer-2 middleware / Layer-3 handler:
   id, ok := serve.IdentityFromContext(sess.Context())
```

The framework's terminal `ConnectHandler` (the innermost no-op) captures the final `*http.Request` it received, and the framework uses *that* request's context as the parent for the Session's context. This means middleware that wants its context updates to be visible to layers 2 and 3 must call `next(r)` with the updated request, which is the standard `net/http` pattern.

## Failure Modes

- **Connect chain returns plain `error` (non-`*ConnectError`)**: framework treats as 500/QUIC 0x02. Logged.
- **Connect chain returns `*ConnectError` with invalid Status (< 100 or > 599)**: framework normalizes to 500.
- **Lifted HTTP middleware writes a response *and* calls next**: undefined; we document "don't do this." The adapter returns whatever next returns; the response was already written.
- **Session middleware constructor panics**: propagates; the Server panics during install. (No recover at install time — bad middleware is a programmer error, not a runtime concern.)
- **Session middleware lifecycle goroutine outlives the Session**: relies on `sess.Done()` for shutdown. Documented contract: middleware MUST select on `sess.Done()` (or `sess.Context().Done()`) to exit. Leaks here are middleware bugs.
- **`Session.Close()` called more than once**: documented as idempotent — subsequent calls return nil. Implementations of the wrapping pattern delegate to the embedded Session's `Close()`, which itself is idempotent.
- **Initial Resize exceeds `MaxWindowDims`**: connection closed without creating a session (treated as a client error; logged at debug level).
- **Subsequent Resize exceeds `MaxWindowDims`**: rejected, no resize applied, debug log entry. Connection stays open.
- **Inbound message exceeds `MaxPasteBytes`**: connection closed (paste-bombing is treated as hostile, not just a client mistake).

## Testing Strategy

### ConnectMiddleware

- Chain ordering: install three middlewares that append to a recorded slice; verify outermost-first call order on the way in and the reverse on the way out.
- Built-in `checkAuth`-as-middleware: existing basic-auth tests continue to pass unchanged. New test: `BasicUsername == ""` does not install the basic-auth middleware (verified by the `WWW-Authenticate` header being absent on a no-config request).
- Built-in connection limit: prior `503` behavior preserved; release-on-disconnect verified.
- `LiftHTTPMiddleware` — wrapped middleware calls `next`: chain proceeds.
- `LiftHTTPMiddleware` — wrapped middleware writes response and skips `next`: framework does not write again; test asserts the response body matches the lifted middleware's output exactly.

### ConnectError

- WS path: returning `&ConnectError{Status, Headers, Body}` writes the expected status, headers, and body.
- WT path: returning `&ConnectError{Status: 401}` closes with QUIC code `0x01`; explicit `WTCode: 0x99` overrides; plain `error` closes with `0x02`.

### SessionMiddleware

- Ordering: three middlewares each appending to a slice on `OutputReader()`; outermost is observed first.
- Idempotent `Close`: calling `sess.Close()` twice returns nil twice.
- Lifecycle goroutine: an activeterm-style mock middleware that closes after 50ms; verify session is closed and no goroutine leaks (via `goleak` or runtime goroutine count).

### Middleware (layer 3)

- Ordering: three middlewares each prepending to the model's view; outermost runs first.
- Recover middleware (test fixture, not shipped): a panicking handler is recovered and the connection is closed cleanly.

### Identity

- Round-trip: install a `ConnectMiddleware` that sets `WithIdentity(ctx, testID{"alice"})`; verify `IdentityFromContext(sess.Context())` returns `{"alice", true}` from inside both a `SessionMiddleware` and a `Handler`.

### Config knobs

- `MaxWindowDims`: initial Resize at 5000x5000 with default cap → connection closed; resize at 100x100 → accepted.
- `MaxWindowDims` mid-session: post-handshake Resize at 5000x5000 → rejected, session continues with prior size.
- `MaxPasteBytes`: paste at 2 MiB with default cap → connection closed.
- `ResizeThrottle`: send 100 resizes within 16ms → only one observed by the underlying PTY (most recent value).

## Forward-Looking: v0.4 Layout

`serve/sipmetrics/` is reserved as the home for v0.4's Prometheus-based session metrics middleware (a `SessionMiddleware`). Subpackage rather than separate module to avoid release coordination, with the cost of a `prometheus/client_golang` dep — opted into by importing the subpackage. v0.3 spec mentions this but ships no code for it.

## Out of Scope for v0.3

- Built-in middleware: `osc52gate`, `activeterm`, `idletimeout`, `recover`, `logging`, `sipmetrics`. All v0.4.
- Named middleware / reorderable chains. v0.3 is "list, in order, immutable post-install."
- Replacing `cfg.BasicUsername` with a richer auth-config story. The current basic-auth surface stays as-is; users wanting more sophisticated auth install their own `ConnectMiddleware`.
- Handshake-response richness beyond `*ConnectError` (e.g., custom QUIC datagram replies). YAGNI.
- A formal `serve.Recipes` doc enumerating example chains. Useful but separate.
